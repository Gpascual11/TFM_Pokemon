\section{Entorn de Simulació}
\label{sec:entorn}

\subsection{Arquitectura Client-Servidor de Pokémon Showdown}

Pokémon Showdown~\cite{pokemon_showdown} és un simulador de combats Pokémon de codi obert que implementa la totalitat de les regles oficials de les generacions 1 a 9. A nivell tècnic, s'estructura com una aplicació \textbf{Node.js} que exposa un servidor de combats accessible via el protocol \textbf{WebSocket} (RFC~6455). Aquesta arquitectura client-servidor és fonamental per entendre com els agents d'IA interactuen amb el joc.

\subsubsection{Llançament del Servidor Local}

En un entorn de recerca, Showdown s'executa localment sense interfície gràfica ni autenticació:

\begin{verbatim}
  node pokemon-showdown start --port 8000 --no-security
\end{verbatim}

El flag \texttt{--no-security} desactiva el sistema d'autenticació (passwords, tokens), permetent que qualsevol client local es connecti directament. Cada instància del servidor ocupa un \textbf{port TCP exclusiu} i funciona com un procés Node.js independent que manté en memòria volàtil (RAM) l'estat de totes les batalles actives. No hi ha persistència a disc: quan el procés es tanca, tot l'estat es perd.

\subsubsection{Protocol de Comunicació WebSocket}

La comunicació entre un agent (client) i el servidor utilitza \textbf{WebSocket}, un protocol full-duplex sobre TCP que manté una connexió persistent. A diferència de HTTP (request-response), WebSocket permet que el servidor enviï missatges al client en qualsevol moment sense necessitat de polling.

Els missatges de Showdown utilitzen un \textbf{format de text pla} amb una sintaxi basada en delimitadors \texttt{|}. Cada línia representa un event del combat:

\begin{verbatim}
  |move|p1a: Pikachu|Thunderbolt|p2a: Gyarados
  |-damage|p2a: Gyarados|23/100
  |-supereffective|p2a: Gyarados
  |turn|5
\end{verbatim}

L'estructura general és \texttt{|tipus\_event|argument1|argument2|...}. Els prefixos \texttt{|-} denoten events secundaris (dany, efectes, estats alterats). Aquesta codificació en text pla és molt lleugera ($\sim$50--200 bytes per torn) comparada amb un format binari o JSON complet.

\subsubsection{El Missatge \texttt{request}: El JSON d'Estat}

El missatge més important per a un agent és el \texttt{|request|}, que el servidor envia a cada jugador al començament de cada torn. Conté un objecte JSON complet amb tota la informació disponible per prendre una decisió:

\begin{verbatim}
  |request|{"active":[{"moves":[
    {"move":"Thunderbolt","id":"thunderbolt","pp":24,
     "maxpp":24,"target":"normal","disabled":false},
    {"move":"Iron Tail","id":"irontail","pp":24,
     "maxpp":24,"target":"normal","disabled":false},
    ...
  ]}],"side":{"pokemon":[
    {"ident":"p1: Pikachu","details":"Pikachu, L84, M",
     "condition":"187/187","active":true,
     "stats":{"atk":137,"def":98,"spa":148,"spd":118,"spe":178},
     "moves":["thunderbolt","irontail","surf","voltswitch"],
     "item":"lightball","ability":"Static"},
    ...
  ]}}
\end{verbatim}

Aquest JSON conté: els moviments disponibles (amb PP restants, si estan desactivats), les estadístiques exactes dels Pokémon propis, l'objecte equipat, l'habilitat, i la condició actual (\texttt{HP\_actual/HP\_màxim status}). Notar que les estadístiques i objectes de l'oponent \textbf{no s'inclouen} mai --- això formalitza la informació imperfecta.

\subsubsection{Resposta de l'Agent: Ordres de Combat}

L'agent respon amb una cadena de text simple que codifica la seva decisió:

\begin{itemize}
    \item \texttt{/choose move 1} --- Usar el primer moviment de la llista.
    \item \texttt{/choose move 3 -2} --- Usar el tercer moviment apuntant al segon oponent (format Doubles).
    \item \texttt{/choose switch 4} --- Substituir pel quart Pokémon de l'equip.
    \item \texttt{/choose move 1 terastallize} --- Atacar amb Teracristal·lització activa.
\end{itemize}

Un cop el servidor rep les ordres d'ambdós jugadors, \textbf{resol el torn de forma atòmica}: aplica les regles de prioritat, compara velocitats, calcula dany amb la fórmula oficial (incloent un factor aleatori entre 0.85 i 1.0), i envia el flux de events resultant a ambdós clients.

Punts importants d'aquest cicle:
\begin{itemize}
    \item Tot l'estat del combat viu exclusivament a la \textbf{RAM del procés Node.js}. No hi ha cap escriptura a disc durant la batalla.

    \item Cada instància del servidor pot gestionar \textbf{múltiples batalles simultànies} de forma concurrent gràcies a l'event loop de Node.js.
\end{itemize}

\label{subsec:pokeenv}

Proporciona una interfície orientada a objectes que permet als investigadors centrar-se en la lògica de decisió sense gestionar connexions de xarxa ni parsejar missatges de text.

\subsubsection{Classes Principals}

\begin{itemize}
    \item \textbf{\texttt{Player}}: Classe base abstracta per a qualsevol agent. El desenvolupador només necessita implementar el mètode \texttt{choose\_move(battle)} que rep l'estat actual i retorna una acció. Poke-env s'encarrega de tota la gestió de connexió, autenticació i sincronització.
    \item \textbf{\texttt{Battle}}: Encapsula l'estat complet observable d'una batalla a cada torn. Conté: \texttt{active\_pokemon}, \texttt{opponent\_active\_pokemon}, \texttt{available\_moves}, \texttt{available\_switches}, \texttt{weather}, \texttt{fields}, \texttt{side\_conditions}, etc.
    \item \textbf{\texttt{Pokemon}}: Objecte amb atributs com \texttt{types}, \texttt{stats}, \texttt{boosts}, \texttt{status}, \texttt{current\_hp\_fraction}, \texttt{moves} (revelats), etc.
    \item \textbf{\texttt{Move}}: Conté \texttt{base\_power}, \texttt{type}, \texttt{accuracy}, \texttt{category} (Physical/Special/Status), \texttt{priority}, i efectes secundaris.
\end{itemize}

\subsubsection{Model de Concurrència: asyncio}

Poke-env utilitza el framework \texttt{asyncio} de Python per gestionar múltiples batalles simultàniament dins d'un sol fil d'execució. El paràmetre \texttt{max\_concurrent\_battles} controla quantes batalles pot mantenir obertes un agent alhora (típicament 10--20). Internament, un \texttt{Semaphore} limita la creació de batalles noves fins que les anteriors finalitzin.

Quan es crida \texttt{player.battle\_against(opponent, n\_battles=100)}, poke-env:
\begin{enumerate}
    \item Obre connexions WebSocket per ambdós jugadors.
    \item Llança el challenge i espera l'acceptació.
    \item Per cada torn, crida \texttt{choose\_move()} i envia la resposta.
    \item Un cop rep \texttt{|win|}, marca la batalla com a finalitzada i allibera el semàfor.
\end{enumerate}

Tot aquest procés és \textbf{no-bloquejant}: mentre una batalla espera la resposta del servidor, altres batalles poden estar enviant les seves ordres.

\subsubsection{Compatibilitat amb OpenAI Gym}

Poke-env també exposa una interfície compatible amb l'estàndard \textit{Gymnasium}~\cite{brockman2016openai}, permetent l'ús directe d'algorismes de Reinforcement Learning (DQN, PPO, A2C). Aquesta interfície transforma cada torn en una tupla \texttt{(observation, reward, done, info)} i gestiona automàticament el cicle de vida de l'entorn.

\subsection{Discrepàncies Tècniques: Poke-env vs Pokechamp}

Durant el desenvolupament, s'ha identificat una dicotomia metodològica entre la versió oficial de \textit{poke-env} i la versió utilitzada pel repositori \textit{Pokechamp}. Aquesta distinció és crítica per a la reproductibilitat:

\begin{itemize}
    \item \textbf{Compatibilitat d'Atributs}: La versió de \textit{Pokechamp} utilitza una branca modificada de la llibreria que accedeix a atributs de batalla (com \texttt{active\_pokemon}) de forma diferent a la versió \texttt{0.11+} oficial. Això ha obligat a crear una capa d'abstracció a \texttt{AgentFactory} per garantir que els agents d'ambdues procedències puguin coexistir en el mateix benchmark.
    \item \textbf{Seguiment Estricte (Strict Tracking)}: Mentre que \textit{Pokechamp} ignora certs errors de protocol de Showdown, \textit{poke-env} oficial llança excepcions quan detecta moviments inesperats. La decisió metodològica ha estat relaxar aquest seguiment (\texttt{strict\_battle\_tracking=False}) per permetre la compatibilitat amb la lògica de tercers i assegurar la continuïtat de les simulacions de llarga durada.
\end{itemize}

\subsection{Necessitat d'Infraestructura Paral·lela}
\label{subsec:necessitat_paralel}

L'execució de benchmarks estadísticament significatius (10.000 partides per parella d'agents) amb un sol servidor resulta inviable per dues limitacions inherents al sistema descrit anteriorment:

\begin{enumerate}
    \item \textbf{Limitació de Node.js}: Cada instància de Showdown utilitza un únic fil d'execució (event loop). Tot i que pot gestionar múltiples batalles concurrentment, el throughput es satura al voltant de 20--30 batalles simultànies per la serialització de la resolució de torns.
    \item \textbf{Acumulació de memòria}: El Garbage Collector de Node.js no allibera completament la memòria fragmentada de batalles finalitzades. Després de milers de batalles, el procés pot créixer de $\sim$80~MB a $>$500~MB, degradant el rendiment i arriscant un \textit{Out of Memory} (OOM).
\end{enumerate}

Per superar-ho, la metodologia d'avaluació d'aquest TFM es basa en un patró \textbf{Master-Worker} amb múltiples instàncies de Showdown executant-se en ports paral·lels. Cada worker és un procés Python independent que es comunica amb el seu propi servidor Showdown dedicat, eliminant qualsevol contenció entre experiments. Els detalls d'implementació d'aquesta arquitectura es descriuen al Capítol~\ref{chap:implementacio} (Secció~5.2).

\subsection{Disseny de la Matriu d'Avaluació (Benchmark Matrix)}

Per tal de garantir una comparació justa i exhaustiva entre els diversos agents, s'ha implementat una metodologia d'\textbf{avaluació per matriu completa}. Un \textit{run} del benchmark no avalua un sol agent, sinó que processa una matriu creuada de $N$ agents contra $M$ oponents, resultant en $N \times M$ matchups independents.

\begin{itemize}
    \item \textbf{Recuperació automàtica (Resume)}: El sistema comprova l'existència de dades prèvies a disc. Si d'un matchup de 10.000 partides ja se n'han completat 2.000 en una execució anterior, el benchmark només llançarà les 8.000 restants. Això fa el procés tolerant a interrupcions (talls de corrent, errors de xarxa, reinicis del sistema).
    \item \textbf{Consistència de Formats}: Tots els agents del benchmark s'avaluen sota exactament el mateix format i generació (p.~ex.\ \texttt{gen9randombattle}), garantint que les diferències de rendiment es deuen exclusivament a la qualitat de la decisió, no a variacions de l'entorn.
    \item \textbf{Aleatorietat controlada}: En el format \textit{Random Battle}, els equips són assignats aleatòriament pel servidor Showdown a cada partida. Amb 10.000 partides per matchup, la llei dels grans nombres garanteix que qualsevol avantatge puntual per equip es dilueix estadísticament.
\end{itemize}

\section{Complexitat del Problema}
\label{sec:complexitat}

\subsection{Diferencies entre Generacions}

Cada generació de jocs de Pokémon defineix un conjunt diferent de regles, espècies disponibles i mecàniques especials. Això impacta directament en la complexitat del problema:

\begin{itemize}
    \item \textbf{Nombre de Pokémon legals}: Varia entre $\sim$150 (Generació~I) i $>$1.000 (Generació~IX).
    \item \textbf{Mecàniques especials}: Mega-Evolucions (Gen~VI--VII), Z-Moves (Gen~VII), Dynamax (Gen~VIII), Teracristal·lització (Gen~IX) afegeixen accions extra i modificadors.
    \item \textbf{Objectes i habilitats}: La base creix a cada generació, augmentant les combinacions possibles.
\end{itemize}

Per gestionar aquesta variabilitat, l'arquitectura del projecte s'ha dissenyat per ser flexible: el format i la generació es passen com a paràmetres configurables, permetent entrenar i avaluar agents en qualsevol combinació.

\subsection{Selecció de Generacions i Verificació de Formats}

El projecte s'ha dissenyat per ser agnòstic a la generació, però l'avaluació real depèn de la disponibilitat de formats al servidor Showdown. Durant la fase d'experimentació, es va dur a terme una auditoria dels formats suportats:

\begin{itemize}
    \item \textbf{Generacions 8 i 9}: Suporten nativament el format \texttt{Random Doubles Battle}, la qual cosa permet una avaluació directa de les heurístiques contra baselines estàndard.
    \item \textbf{Generació 7}: Tot i ser una generació d'alt interès competitiu, es va descobrir que el format oficial de \textit{Random Doubles} no està habilitat per defecte en els servidors locals de Showdown (estant limitat a Singles).
\end{itemize}

Aquesta limitació tècnica va portar a la decisió metodològica de centrar el gruix de l'estudi en les **Generacions 8 i 9**, on les dades són directament comparables amb l'ecosistema competitiu actual.

La diferència de complexitat entre formats és considerable:

\begin{table}[h]
    \centering
    \begin{tabular}{|l|c|c|}
    \hline
    \textbf{Dimensió} & \textbf{Singles (1v1)} & \textbf{Doubles (2v2)} \\ \hline
    Pokémon al camp & 1 propi, 1 rival & 2 propis, 2 rivals \\ \hline

    Objectius d'atac & 1 (el rival) & 3 (rival 1, rival 2, aliat) \\ \hline

El format de \textbf{Doubles} multiplica exponencialment l'espai de decisions: cada jugador ha de coordinar les accions de dos Pokémon simultàniament, tenint en compte sinergies, atacs d'àrea, proteccions i l'interacció entre els quatre Pokémon al camp. Per aquesta raó, el projecte aborda primer el format Singles com a prova de concepte i després escala al format Doubles.

\section{Espai d'Estats i Espai d'Accions}
\label{sec:espais}

Tot problema de RL ha d'estar encapsulat formalment per poder ser processat per un algorisme.

\subsection{Espai d'Estats (S)}

El conjunt $S$ agrupa tota la informació observable per l'agent en un torn donat. Això constitueix un entorn parcialment observable (POMDP), on l'agent només accedeix a una observació $\mathbf{O}DUMMY\_UNDERSCOREt \in \mathbb{R}^d$ que resumeix:

\begin{itemize}
    \item Vida (HP) restant de tots els Pokémon visibles (propis i rivals).
    \item Modificadors d'estadístiques actius (\textit{boosts}).
    \item Condicions d'estat (status) actives.
    \item Informació de camp (clima, terreny, hazards).
    \item Historial acumulat de moviments i tipus observats del rival.
\end{itemize}

En el format de Singles, el vector d'observació resultant té de l'ordre de $d \sim 50\text{--}100$ dimensions. En Doubles, es duplica aproximadament per la presència de dos Pokémon actius per bàndol.

\subsection{Espai d'Accions (A)}

L'espai d'accions $A$ conté els moviments i substitucions legals disponibles:

\begin{itemize}
    \item \textbf{Singles}: Fins a 4 atacs + fins a 5 switches = màxim 9 accions discrets per torn.
    \item \textbf{Doubles}: L'agent ha d'escollir una acció per a \textit{cadascun} dels dos Pokémon al camp. A més, cada atac ha d'especificar l'objectiu (\textit{target}): rival~1, rival~2, o aliat. Això genera un producte cartesià de possibilitats que pot arribar a desenes de combinacions legals per torn.
\end{itemize}

Cal notar que l'espai d'accions és \textbf{dinàmic}: varia a cada torn segons els Pokémon vius, els PP restants, si un Pokémon ha estat derrotat i obliga un switch forçós, etc.


% ============================================================
%  2.4 — MÈTRIQUES D'AVALUACIÓ
% ============================================================
\section{Mètriques d'Avaluació}
\label{sec:metriques}

La naturalesa estocàstica del combat Pokémon invalida qualsevol avaluació basada en poques partides: un sol resultat pot estar dominat per l'atzar. Per tant, totes les comparacions entre agents es realitzen sobre sèries massives de batalles ($\geq 100$ per parella).

\subsection{Mètriques Primàries}

\begin{itemize}
    \item \textbf{Win-Rate (\%)}: Percentatge de victòries de l'agent sobre el total de batalles disputades contra un oponent concret. És la \textbf{mètrica principal i definitiva} del projecte a l'hora de jutjar si un agent és superior a un altre; totes les altres mesures s'interpreten com a suport qualitatiu a aquesta.
    \begin{itemize}
        \item $\sim 50\%$: L'agent no millora respecte un rival aleatori simètric.
        \item $> 60\%$: Avantatge significatiu; l'agent demostra intel·ligència estratègica.
        \item $> 85\%$: Domini clar; l'agent supera àmpliament l'oponent.
        \item $100\%$: Intrínsecament impossible en un entorn estocàstic com Pokémon.
    \end{itemize}
\end{itemize}

\subsection{Mètriques Secundàries}

Per complementar el Win-Rate i obtenir una visió més granular del comportament dels agents, es recullen mètriques addicionals:

\begin{itemize}
    \item \textbf{Durada de les partides (torns)}: Un agent eficient tendeix a guanyar en menys torns. Partides molt llargues poden indicar indecisió o incapacitat d'aprofitar avantatges.
    \item \textbf{Pokémon derrotats (propis)}: Quantes unitats ha perdut l'agent durant la partida. Un nombre baix indica millor gestió defensiva.
    \item \textbf{Pokémon derrotats (rivals)}: Quantes unitats rivals ha eliminat l'agent. Pot diferenciar entre victòries ajustades i dominants.
    \item \textbf{Pokémon restants vius}: Tant propis com rivals al final de la partida. Indica el marge de victòria o derrota.
    \item \textbf{Vida total restant (\%)}: Suma de la vida percentual de tots els Pokémon supervivents (propis i rivals). Aporta una mesura contínua i més granular que el simple recompte de Pokémon vius.
\end{itemize}

% ============================================================
%  2.5 — DISSENY DELS EXPERIMENTS
% ============================================================
\section{Disseny dels Experiments}
\label{sec:disseny_experiments}

Amb l'entorn de simulació i les mètriques definides, cal especificar com s'utilitzen per avaluar els diferents agents. Sense entrar encara en els resultats numèrics (Capítol~\ref{chap:resultats}), aquesta secció descriu l'esquema general del disseny experimental.

\subsection{Tipus de Matchups}

Els experiments es basen en sèries de batalles entre parelles d'agents $(A, B)$, on cada agent pot ser:
\begin{itemize}
    \item Un agent heurístic propi (versions v1--v12 per Singles).
    \item Un agent de Reinforcement Learning entrenat amb el pipeline descrit al Capítol~\ref{chap:implementacio}.
    \item Un baseline natiu de \textit{poke-env} (RandomPlayer, MaxBasePowerPlayer, SimpleHeuristicPlayer).
    \item Un agent extern del repositori Pokechamp (Abyssal, SafeOneStep, Pokechamp, PokeLLMon), quan el format i generació són compatibles.
\end{itemize}

Per a cada parella $(A, B)$ es defineixen \textbf{sèries de com a mínim 100 batalles}, amb equips inicials aleatoris dins del format triat (p.\ ex.\ \texttt{gen9randombattle} per a Singles). L'ordre de sortida (qui comença com a jugador 1 o 2) es balanceja perquè l'avantatge inicial no es barregi amb la qualitat de l'agent.

\subsection{Formats i Condicions}

Tot i que el projecte està dissenyat per donar suport a múltiples generacions i formats, els experiments d'aquest TFM se centren principalment en:
\begin{itemize}
    \item \textbf{Singles (1 vs 1)} com a banc de proves principal per a heurístiques i RL.
    \item La generació i el format de Random Battles adequats per a la integració amb Pokechamp (Gen~9 Singles), quan s'inclouen aquests agents externs.
\end{itemize}

Les condicions d'entorn (clima, hazards, objectes, habilitats) són les definides pel simulador oficial de \textit{Showdown} per al format escollit; el TFM no modifica les regles del joc, sinó que s'hi adapta.

\subsection{Agregació de Resultats}

Per a cada sèrie de batalles es calcula:
\begin{itemize}
    \item El \textbf{Win-Rate} de cada agent, que és la mètrica principal utilitzada per comparar-los.
    \item Les mitjanes i distribucions de les mètriques secundàries descrites a la Secció~\ref{sec:metriques}, que es faran servir al Capítol~\ref{chap:resultats} per interpretar qualitativament el comportament dels agents (eficiència temporal, agressivitat, solidesa defensiva, etc.).
\end{itemize}

Aquest disseny experimental garanteix que qualsevol diferència de rendiment observada en els resultats sigui estadísticament fiable i no deguda a la variabilitat inherent de poques partides.

\subsection{Propera Fase: Anàlisi de la Distribució de Pokémon}
\label{subsec:next_phase_dist}

Un dels reptes immediats un cop finalitzada l'avaluació massiva és l'anàlisi de les dades recollides. Tot i que els equips a \textit{Random Battles} són generats aleatòriament pel servidor Showdown, aquests segueixen distribucions de probabilitat internes. 

Està previst realitzar un estudi estadístic sobre el \textit{usage-rate} i el \textit{win-rate} individual de cada espècie de Pokémon dins d'aquests formats aleatoris. Aquesta anàlisi permetrà identificar si certs Pokémon tenen un impacte desproporcionat en la victòria independentment de l'agent que els controli, proporcionant una base per a una futura "normalització" dels resultats segons la qualitat de l'equip rebut.

% ============================================================
%  2.6 — FASES DE DESENVOLUPAMENT
% ============================================================
\section{Fases de Desenvolupament i Models de Referència}
\label{sec:fases}

El desplegament d'agents sobre \textit{Showdown} s'ha estructurat metodològicament en dues grans fases consecutives per garantir una monitorització i isolació de variables controlada. 

\subsection{Fase 1: Disseny de la Lògica de Decisió Heurística}

Abans de procedir amb arquitectures d'aprenentatge automàtic, s'ha dissenyat un sistema d'heurístiques expertes. L'objectiu no és només crear un agent fort, sinó \textbf{formalitzar matemàticament el coneixement clàssic del combat Pokémon} per servir com a base de dades i mètrica de comparació (\textit{Baseline}).

\subsubsection{Metodologia Heurística 1vs1 (Singles)}
La lògica de decisió s'ha dividit en mòduls incrementals que l'agent avalua a cada torn:
\begin{itemize}
    \item \textbf{Estadística i Dany}: Càlcul de dany estimat basat en la formula oficial, ponderant Atac/Defensa, bonificació per tipus (STAB) i resistències de l'oponent.
    \item \textbf{Consciència d'Estat (Utility)}: Assignació de pesos extra a moviments que generen estats alterats (Burn, Paralysis) segons la vulnerabilitat del rival.
    \item \textbf{Lògica de Pivot (Switching)}: Metodologia per decidir quan un "canvi" és millor que un "atac". Això es formalitza avaluant si el dany rebut en el proper torn serà superior a un llindar crític i si existeix un aliat al banc amb una resistència de tipus favorable.
\end{itemize}

\subsubsection{Metodologia Heurística 2vs2 (Doubles)}
El format de Dobles introdueix el problema de la \textbf{Coordinació Espacial}. No n'hi ha prou amb triar la millor acció per a cada slot individualment; cal que ambdues accions maximitzin la sinergia global. S'ha implementat el patró \textbf{"Score-then-Combine"}:

\begin{enumerate}
    \item \textbf{Fase de Scoring}: Es puntua cada possible moviment d'atac o canvi per als dos Pokémon per separat.
    \item \textbf{Fase de Combinació}: L'agent avalua totes les combinacions possibles (fins a 81 en alguns torns) i aplica regles de sinergia:
    \begin{itemize}
        \item \textbf{Focus Fire}: Si ambdós Pokémon ataquen el mateix objectiu, reben un bonus si la suma de dany garanteix un KO (evitant el malbaratament d'atacs).
        \item \textbf{Protecció Selectiva}: Si un Pokémon està en una posició vulnerable (poca vida o desavantatge de tipus), el sistema avalua si "Protegir-se" mentre l'aliat ataca és més rendible que atacar directament.
        \item \textbf{Mitigació de Residus}: Penalització si ambdós Pokémon ataquen a un rival que moriria amb un sol atac, incentivant que el segon Pokémon ataqui a l'altre oponent.
    \end{itemize}
\end{enumerate}

\subsection{Fase 2: Models d'Aprenentatge Profund i Raonament per LLM}

Una vegada validada l'arquitectura i disposant del \textit{Baseline Expert}, la segona gran fase de la recerca se centra en la substitució de la codificació dura per arquitectures d'aprenentatge o raonament.

\subsubsection{Raonament per Models de Llenguatge (LLM)}
La inclusió de LLMs com \textit{Qwen 8B} respon a una hipòtesi de recerca: pot un model de llenguatge generalista aplicar heurístiques de combat sense haver estat entrenat exclusivament per a Pokémon? Metodològicament, això s'aborda mitjançant:
\begin{itemize}
    \item \textbf{Inferència Local amb Ollama}: Per evitar el biaix de latència i el cost de les APIs de tercers, s'utilitza \textit{Ollama} com a backend local. Això permet que el raonament de l'agent sigui privat, cost-zero i altament reproducible.
    \item \textbf{Chain of Thought (CoT)}: No només es demana l'acció final, sinó que s'incentiva que el model "pensi" en veu alta (Strategy Logs). Això permet una avaluació qualitativa de la metodologia: l'agent ha triat el moviment correcte pel motiu correcte?
\end{itemize}

\subsubsection{Reinforcement Learning (PPO)}
L'eix científic del TFM assoleix el seu màxim pragmatisme avaluant si algorismes com PPO poden descobrir per si sols les normes causals que la Fase~1 programava explícitament (Taula de Tipus, STAB, predicció de canvis).

\subsection{Línies de Control (Baselines)}

Amb l'entorn escalable definit, cal fixar les línies base de referència sobre les quals mesurar tot progrés. S'utilitzen dos conjunts de baselines complementaris.

\paragraph{Baselines natius de poke-env.}
Dins de poke-env~\cite{poke_env} es proporcionen tres agents de referència que funcionen en \textbf{qualsevol generació i format} (Singles i Doubles):

\begin{enumerate}
    \item \textbf{\textit{RandomPlayer}}: Efectua moviments a l'atzar entre les accions legals. Actua com a referència mínima: qualsevol agent amb un mínim d'intel·ligència hauria de superar-lo àmpliament.
    \item \textbf{\textit{MaxBasePowerPlayer}}: Un agent cobdiciós (\textit{Greedy}) que escull sempre l'atac amb la potència base més alta, ignorant tipus, STAB i context defensiu. Representa la barrera inferior d'estratègia funcional.
    \item \textbf{\textit{SimpleHeuristicPlayer}}: El rival més sofisticat ofert per la llibreria. Incorpora heurístiques preprogramades: avalua debilitats de tipus, prioritza canvis (\textit{switches}) quan el seu Pokémon actiu es troba en desavantatge, i maximitza el dany considerant matchups. Superar aquest agent de manera consistent certifica que les noves funcions pròpies han assolit una intel·ligència superior.
\end{enumerate}

\paragraph{Baselines externs de Pokechamp.}
Des del repositori Pokechamp~\cite{pokechamp} s'han integrat agents addicionals, principalment orientats al format \textbf{Singles de Generació~9} (\texttt{gen9randombattle}):

\begin{enumerate}
    \item \textbf{\textit{AbyssalPlayer}}: Agent heurístic avançat que avalua matchups de tipus, estima el dany esperat i decideix substitucions basant-se en l'anàlisi de tot l'equip rival. Incorpora lògica específica de Gen~9 (Teracristal·lització). Funciona com a rèplica millorada del SimpleHeuristicPlayer amb coneixement generacional.
    \item \textbf{\textit{SafeOneStepPlayer}}: Agent amb estimació de dany a un torn vista (\textit{one-step lookahead}): puntua cada moviment per $\text{potència} \times \text{STAB} \times \text{efectivitat} \times \text{precisió}$ i escull el millor. S'ha reimplementat internament per evitar dependències del simulador local de Pokechamp que causaven bloquejos.
    \item \textbf{\textit{Pokechamp i PokeLLMon (LLM)}}: Agents basats en Models de Llenguatge (descrits al Capítol~\ref{chap:marc_teoric}) que utilitzen raonament textual per prendre decisions. Requereixen un backend LLM local (detalls d'implementació al Capítol~\ref{chap:implementacio}). Aplicables exclusivament a \textbf{Singles Gen~9}.
\end{enumerate}

El contrast sistemàtic contra \textbf{ambdós conjunts} de baselines ---~natius i externs~--- constitueix el filtre fonamental de la metodologia, a través del qual qualsevol avanç real serà rigorosament validat. Els capítols d'Implementació (Cap.~\ref{chap:implementacio}) i de Resultats (Cap.~\ref{chap:resultats}) despleguen com s'han implementat aquests agents i quins avantatges o limitacions mostren respecte els models proposats en aquest TFM.
