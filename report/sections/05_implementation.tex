\section{Contribucions de Software del Projecte}

Abans de descriure el funcionament intern de cada component, es resumeixen les principals peces de software desenvolupades específicament en el marc d'aquest TFM:

\begin{itemize}
    \item \textbf{Orquestrador paral·lel de servidors Showdown}: Scripts a \texttt{src/p03\_scripts/} per llançar múltiples instàncies locals de Showdown, distribuir batalles de forma asíncrona i gestionar \textit{checkpoints} i reinicis controlats.
    \item \textbf{Família d'heurístiques internes per Singles}: Agents a \texttt{src/p01DUMMY\_UNDERSCOREheuristics/s01\_singles/agents/internal/} (v1--v6) que implementen estratègies cada vegada més sofisticades basades en Taula de Tipus, STAB, status, boosts, prioritat i regles de \textit{switching} defensiu.
    \item \textbf{Heurístiques preliminars per Dobles VGC}: Arquitectura \textit{Score-then-Combine} per al format 2 vs 2, situada a \texttt{src/p01DUMMY\_UNDERSCOREheuristics/s02DUMMY\_UNDERSCOREdoubles/}, actualment en fase de finalització.
    \item \textbf{Reimplementació de SafeOneStepPlayer}: Versió pròpia de l'agent \textit{OneStepPlayer} de Pokechamp, que elimina dependències de simuladors locals i opera exclusivament sobre l'API de \textit{poke-env}.
    \item \textbf{Pipeline complet de Reinforcement Learning}: Mòduls a \texttt{src/p02\_rl\_models/} que inclouen el \texttt{StateDUMMY\_UNDERSCOREVectorizer}, l'entorn \texttt{PokemonMaskedEnv} amb \textit{action masking}, la funció de recompensa i la configuració de \textit{MaskablePPO}.
    \item \textbf{Infraestructura de \textit{logging} i benchmarking}: Escriptures sistemàtiques de resultats numèrics i, en el cas dels agents LLM, logs de raonament textual per a anàlisi posterior.
\end{itemize}

Aquests components es combinen amb llibreries i agents existents (poke-env, Pokechamp, sb3-contrib) que es fan servir com a fonaments sobre els quals construir i avaluar les contribucions pròpies.

\section{Arquitectura Master-Worker per a Benchmarking Massiu}
\label{sec:master_worker}

Com s’ha introduït a la Secció~\ref{subsec:necessitat_paralel}, l’execució de 10.000 partides per matchup requereix una infraestructura paral·lela robusta. Aquesta secció detalla l’arquitectura implementada, les decisions de disseny i els mecanismes de tolerància a fallades.

\subsection{Visió General del Patró}

El sistema segueix un patró \textbf{Master-Worker} clàssic adaptat a les particularitats del simulador Pokémon Showdown:

\begin{itemize}
    \item \textbf{Master} (\texttt{benchmark.py}): Procés Python principal que itera sobre la matriu de matchups, gestiona la cua de ports disponibles, llança workers i consolida resultats.
    \item \textbf{Workers} (\texttt{worker.py}): Subprocessos Python efímers, cadascun amb el seu propi event loop \texttt{asyncio}, que executen un lot de batalles contra un servidor Showdown dedicat.
    \item \textbf{Servidors Showdown}: Instàncies Node.js independents, cadascuna escoltant en un port TCP exclusiu (8000, 8001, ..., 800N).
\end{itemize}

\subsection{Paral·lelisme i Thread-Safety}
\label{subsec:thread_safety}

La decisió d’utilitzar \textbf{multiprocessing} en lloc de \textbf{threading} és fonamental i respon a tres restriccions tècniques:

\begin{enumerate}
    \item \textbf{Global Interpreter Lock (GIL)}: El GIL de CPython impedeix l’execució paral·lela real de codi Python en múltiples fils. Threading només aporta beneficis per a operacions I/O-bound, però el càlcul de les heurístiques és CPU-bound.
    \item \textbf{Aïllament d’estat asyncio}: Poke-env utilitza internament un event loop \texttt{asyncio} per gestionar les connexions WebSocket. Si s’utilitza \texttt{fork()} (el mode per defecte de \texttt{multiprocessing} en Linux), l’estat de l’event loop es copia al fill, causant \textit{deadlocks}. Per evitar-ho, s’utilitza el context \texttt{spawn}:
    \begin{verbatim}
  _mp_ctx = multiprocessing.get_context("spawn")
    \end{verbatim}
    Això crea un intèrpret Python completament nou per cada worker, sense herència d’estat.
    \item \textbf{Alliberament garantit de memòria}: Quan un subprocés acaba (exit code 0 o error), el sistema operatiu reclama \textbf{tota} la RAM que ocupava. Això elimina qualsevol \textit{memory leak} acumulat per poke-env, objectes \texttt{Battle}, o models LLM carregats en VRAM.
\end{enumerate}

Cada worker opera sobre un port Showdown \textbf{exclusiu}. No hi ha cap recurs compartit entre workers: ni memòria, ni connexions de xarxa, ni fitxers (cada un escriu al seu propi \texttt{\_tmp\_*.csv}). Això fa el sistema inherentment \textit{thread-safe} sense necessitat de locks, semàfors o sincronització explícita.

\textbf{Dins de cada worker}, el cicle de gestió de memòria és:
\begin{enumerate}
    \item Es juguen 25 batalles simultàniament (chunk) mitjançant \texttt{battle\_against()}.
    \item S’extreuen les mètriques de les batalles finalitzades i s’escriuen al CSV parcial.
    \item Es crida \texttt{player.reset\_battles()} i \texttt{opponent.reset\_battles()} per eliminar les referències als objectes \texttt{Battle} acumulats.
    \item Es crida \texttt{gc.collect()} per forçar la recollida d’objectes orfes.
    \item Es repeteix fins a completar el lot assignat.
\end{enumerate}

Sense aquest cicle, 10.000 batalles acumulades en un sol procés consumirien $>$2~GB de RAM. Amb el chunking, el consum es manté estable al voltant de $\sim$200--400~MB per worker.

\subsection{Tolerància a Fallades (Crash-Safe State)}
\label{subsec:crash_safe}

L’execució de desenes de milers de partides pot trigar hores. El sistema implementa múltiples mecanismes per garantir que cap fallada causa la pèrdua del treball realitzat:

\subsubsection{Recuperació Automàtica (Resume)}

A l’inici de cada matchup, el master llegeix el CSV de sortida i compta les partides existents:

\begin{verbatim}
  already_done = len(pd.read_csv(out_csv))
  n_to_run = target_battles - already_done
  if n_to_run <= 0: return  # Matchup ja complet
\end{verbatim}

Si l’execució es va interrompre (tall de corrent, error de xarxa, reinici del sistema), simplement re-executar el benchmark reprèn exactament on es va quedar. No cal cap flag \texttt{--resume} ni fitxers de checkpoint: la veritat rau en el CSV.

\subsubsection{Aïllament de Fallades per Worker}

Si un worker crasha (timeout, error de connexió, OOM):
\begin{itemize}
    \item El seu fitxer temporal \texttt{\_tmp\_*.csv} es neteja sempre (bloc \texttt{finally}).
    \item El CSV principal \textbf{no es corromp}: les dades només s’hi afegeixen (\textit{append}) un cop el worker ha completat satisfactòriament.
    \item El master detecta la falta de progrés i pot re-assignar les partides pendents en la següent iteració del bucle de reintents.
\end{itemize}

\subsection{Reinici Periòdic de Servidors}
\label{subsec:restart_logic}

S’ha observat empíricament que els processos Node.js de Pokémon Showdown experimenten una degradació gradual de rendiment:

\begin{itemize}
    \item \textbf{Causa}: El Garbage Collector de V8 (motor JavaScript) utilitza un algoritme generacional. Objectes de llarga durada (metadades de formats, estadístiques de Pokémon carregades en memòria) es promouen a l’\textit{old generation}, on la recollida és menys agressiva. Això causa fragmentació de memòria que mai es recupera completament.
    \item \textbf{Símptoma}: Després de $\sim$3.000--5.000 batalles, el procés Node.js creix de $\sim$80~MB a $>$400~MB i la latència per torn augmenta un $\sim$30\%.
    \item \textbf{Solució}: Reinici periòdic controlat pel master.
\end{itemize}

El reinici s’activa cada N matchups \textbf{actius} (configurable amb \texttt{--restart-every}, per defecte 3). El procediment és:

\begin{enumerate}
    \item \texttt{pkill -f pokemon-showdown} --- Mata tots els processos Showdown.
    \item Sleep 2 segons --- Permet l’alliberament dels ports TCP.
    \item Relançament de N instàncies noves via script bash.
    \item Sleep 15 segons --- Temps d’inicialització (càrrega de JSONs de Pokémon, moviments i habilitats a RAM).
\end{enumerate}

El cost total és de $\sim$17 segons de \textit{downtime} per reinici. En una execució típica de 4 hores amb 30 matchups, això suposa menys de 3 minuts acumulats ($\sim$1\% del temps total), un cost negligible a canvi de la garantia d’estabilitat.

\subsection{Instrumentació: StatsBattle}

Per recollir les mètriques avançades (crits, misses, moviments per Pokémon), s’ha implementat una subclasse \texttt{StatsBattle} que intercepta els missatges WebSocket de Showdown abans que siguin processats per poke-env. Això s’aconsegueix mitjançant un \textit{monkey-patch} dinàmic:

\begin{enumerate}
    \item Al carregar-se, el worker substitueix el mètode \texttt{Player.\_create\_battle()} per una versió que instancia \texttt{StatsBattle} en lloc de la classe base \texttt{Battle}.
    \item \texttt{StatsBattle} sobreescriu \texttt{parse\_message()} per detectar events com \texttt{|-crit|}, \texttt{|-miss|} i \texttt{|-supereffective|} i actualitzar comptadors interns.
    \item Sobreescriu \texttt{switch()} per distingir entre canvis voluntaris (l’agent decideix canviar) i forçats (un Pokémon ha caigut i s’ha de triar substitut).
    \item Al final de la batalla, els comptadors es serialitzen directament al CSV sense cap overhead addicional.
\end{enumerate}

Aquesta tècnica és transparent per als agents: no necessiten cap modificació per funcionar amb \texttt{StatsBattle}, ja que la interfície de \texttt{Battle} es manté idèntica.

\section{Fase 1.1: Desenvolupament d'Heurístiques per Singles (1vs1)}

Abans de desplegar models d'aprenentatge, el projecte exigeix validar la infraestructura dins d'un espai d'accions reduït. El format \textbf{Singles} (1vs1) serveix com a arena de desenvolupament: l'agent interactua contra un sol oponent visible, amb un màxim de 9 accions per torn (4 atacs + 5 substitucions). El resultat final d'aquesta fase és un agent heurístic intern ``expert'' que actua com a línia de control pròpia.

Mitjançant sis cicles evolutius de codi, s'han consolidat heurístiques progressivament més sofisticades sota \texttt{src/p01DUMMY\_UNDERSCOREheuristics/s01\_singles/agents/internal/}. A continuació es resumeix aquesta evolució, entenent-la com un procés iteratiu de millora gradual:

\begin{itemize}
    \item \textbf{Heurística v1 (Naïve Aggressive)}: Selecciona el moviment amb la potència base més alta, ignorant completament tipus, STAB i substitucions.
    \item \textbf{Heurística v2 (Damage Awareness)}: Introdueix el càlcul de dany estimat, multiplicant la potència base pel factor d'efectivitat de tipus ($\times 0.5$, $\times 1$, $\times 2$, $\times 4$) i l'STAB ($\times 1.5$).
    \item \textbf{Heurística v3 (Defensive Pivot)}: Afegeix la primera lògica defensiva. Si el Pokémon actiu està enverinat (Toxic) durant més de 2 torns, es substitueix. Si l'oponent és més ràpid i l'atac previst fa menys de 20 HP de dany, es busca un company millor. Aquesta lògica de \textit{switching} és tan estable que les versions posteriors (v4--v6) la reutilitzen.
    \item \textbf{Heurística v4 (Field-Aware)}: Refina el càlcul de dany incorporant el sistema de categories (físic vs. especial, considerant si l'atacant està cremat) i els modificadors climàtics i de terreny (Sol, Pluja, Electric/Grassy/Psychic Terrain).
    \item \textbf{Heurística v5 (Boost-Aware)}: Estén v4 comptant les modificacions d'estadístiques en combat (\textit{boosts}): considera els $+/-$ d'Atac, Defensa i Velocitat en la fórmula de dany, i afegeix regles de seguretat (substitucions preventives quan la vida pròpia és baixa).
    \item \textbf{Heurística v6 (Priority Master)}: Integra totes les regles anteriors i afegeix un multiplicador de $\times 1.2$ als moviments de prioritat (\textit{Quick Attack}, \textit{Shadow Sneak}, etc.), permetent assegurar KOs contra oponents amb poca vida abans de rebre un atac.
\end{itemize}

La versió v6 constitueix la culminació d'aquesta família d'heurístiques i és la que es fa servir com a \textbf{baseline expert propi} en els experiments descrits al Capítol~\ref{chap:metodologia}.

\end{itemize}

\subsection{Agents LLM: Pokechamp i PokeLLMon}
\label{subsec:llm_impl}

Els agents basats en Models de Llenguatge (LLMs) representen un enfocament radicalment diferent: en lloc de regles codificades, utilitzen raonament en text natural per escollir accions.

\paragraph{Backend: Ollama amb Qwen 8B.}
Per executar els LLMs de forma local sense dependre d'APIs costoses, s'ha configurat \textbf{Ollama}~\cite{ollama_2023} com a servidor d'inferència local. El model seleccionat, \textbf{Qwen 8B}~\cite{qwen_2024}, ocupa $\sim$6--8~GB de VRAM i encaixa completament dins d'una GPU RTX 2080 (8~GB), permetent una inferència ràpida ($\sim$2--5 segons per torn). Models més grans (13B+) causaven \textit{memory swapping} que alentia cada torn a minuts.

La configuració al benchmark s'especifica com:
\begin{verbatim}
  --playerDUMMY\_UNDERSCOREbackend ollama/qwen3:8b
  --playerDUMMY\_UNDERSCOREpromptDUMMY\_UNDERSCOREalgo io
  --temperature 0.3
\end{verbatim}

\paragraph{Arquitectura d'integració.}
L'agent Pokechamp rep l'estat de la batalla en format text, l'envia al LLM via API HTTP, i interpreta la resposta per generar una ordre de batalla. El flux és:
\begin{enumerate}
    \item El benchmark llança un \textit{worker subprocess} per cada batch de batalles.
    \item L'\texttt{AgentFactory} detecta l'identificador (\texttt{pokechamp} o \texttt{pokellmon}) i importa dinàmicament \texttt{get\_llm\_player} del repositori Pokechamp clonat.
    \item L'agent tradueix l'estat de batalla a un \textit{prompt} textual i l'envia a Ollama.
    \item El LLM genera un raonament (\textit{Chain of Thought}) i una acció seleccionada.
    \item L'acció es tradueix a una ordre de Showdown.
\end{enumerate}

\paragraph{Logging de raonament.}
A diferència de les heurístiques, els agents LLM generen fitxers de \textit{``Thinking''} que expliquen el raonament darrere de cada decisió. Aquests logs es guarden a \texttt{evaluation/results/LLM/} i permeten analitzar la qualitat del raonament estratègic del model (anàlisi de tipo matchups, predicció de moviments rivals, gestió de recursos, etc.).

\paragraph{Aïllament de processos.}
El principal repte tècnic va ser la gestió de memòria: el fil de fons \texttt{POKEDUMMY\_UNDERSCORELOOP} de Pokechamp manté referències als objectes de la batalla fins i tot després de \texttt{del} + \texttt{gc.collect()}, acumulant memòria de forma il·limitada. La solució va ser l'\textbf{aïllament per subprocessos}: cada batch de batalles s'executa en un procés Python independent (\texttt{\texttt{worker.py}}), i quan el procés acaba, el sistema operatiu reclama \textit{tota} la memòria.

\section{Fase 2: Implementació del Pipeline de Reinforcement Learning}
\label{sec:rl_impl}

Amb les heurístiques com a baselines experts, la segona gran fase del projecte ha consistit a construir un pipeline complet de Reinforcement Learning per entrenar agents neuronals, implementat a \texttt{src/p02\_rl\_models/}. Aquest pipeline materialitza, en codi, els conceptes de POMDP i PPO introduïts al Capítol~\ref{chap:marc_teoric}: l'agent opera sobre observacions vectoritzades parcials de l'estat i actualitza una política estocàstica restringida per \textit{clipping}.

\subsection{Entorn Gymnasium amb Action Masking}

L'entorn \texttt{PokemonMaskedEnv} hereta de la classe \texttt{SinglesEnv} de poke-env i implementa la interfície estàndard de \textbf{Gymnasium}. L'espai d'accions és discret amb 10 accions possibles:
\begin{itemize}
    \item \textbf{Accions 0--3}: Atacs (un per cada slot de moviment).
    \item \textbf{Accions 4--9}: Substitucions (un per cada posició del banc).
\end{itemize}

Per evitar que l'agent intenti accions il·legals (atacar amb un moviment sense PP, substituir per un Pokémon debilitat), s'utilitza \textbf{Action Masking}: cada torn, un vector binari indica quines de les 10 accions són legals. L'algorisme \texttt{MaskablePPO}~\cite{sb3_contrib} utilitza aquesta màscara per assignar probabilitat zero a les accions invàlides, reduint l'espai efectiu de cerca i estabilitzant l'aprenentatge en un entorn amb moltes accions contextualment prohibides.

\subsection{Reward Shaping (Disseny de la Recompensa)}
\label{subsec:reward}

La funció de recompensa densa combina múltiples senyals per guiar l'aprenentatge:

\begin{itemize}
    \item \textbf{Base}: Delta de HP (propis i rivals) + bonus per Pokémon derrotat $(\pm 2)$ + victòria/derrota $(\pm 30)$.
    \item \textbf{Boosts ofensius}: $+0.3$ per nivell de boost positiu en Atac, Atac Especial o Velocitat.
    \item \textbf{Penalització defensiva}: $-0.1$ per nivell de debuff propi.
    \item \textbf{Status oponent}: $+0.3$ si el rival actiu té un status debilitant (Burn, Paralysis, Toxic).
    \item \textbf{Hazards}: $+0.5$ per hazard plantat al camp rival; penalització proporcional per hazards propis.
    \item \textbf{Penalització temporal}: $-0.02$ per torn, per evitar partides infinitament llargues.
\end{itemize}

\subsection{Arquitectura del Model i Infraestructura GPU}

L'agent utilitza \textbf{MaskablePPO} de la llibreria \texttt{sb3--contrib}~\cite{sb3_contrib} amb els següents hiperparàmetres:
\begin{itemize}
    \item \textbf{Xarxa}: MLP amb 2 capes ocultes de 256 neurones cadascuna (\texttt{netDUMMY\_UNDERSCOREarch=[256, 256]}).
    \item \textbf{Learning rate}: $3 \times 10^{-4}$.
    \item \textbf{Steps per batch}: 2048.
    \item \textbf{Entropy coefficient}: 0.01 (incentiva l'exploració en les primeres fases).
    \item \textbf{Discount factor ($\gamma$)}: 0.99.
\end{itemize}

L'entrenament s'executa en una màquina amb una \textbf{GPU RTX 2080} (8~GB VRAM) i 16 nuclis de CPU. Per maximitzar el throughput, es despleguen \textbf{10 instàncies paral·leles} de Showdown (ports 8000--8009) amb \texttt{SubprocVecEnv}, permetent recopilar experiència 10 vegades més ràpid que amb un sol servidor.

\section{Treball Futur: Aprenentatge Basat en Datasets de Partides}
\label{sec:dataset_training}

Més enllà del Reinforcement Learning pur implementat en aquest TFM, una línia de treball futur prometedora consisteix a explotar \textbf{datasets de partides existents} per entrenar agents mitjançant tècniques d'\textit{Imitation Learning} o \textit{Behavioral Cloning}. Aquesta secció descriu una proposta que \textbf{no s'ha implementat} en el marc d'aquest treball, però que es considera una extensió natural de l'arquitectura actual.

El repositori Pokechamp~\cite{pokechamp} inclou internament un conjunt de replays de partides competitives que registren seqüències completes d'estats i accions. La idea seria:
\begin{enumerate}
    \item \textbf{Extracció}: Parsejar els replays per obtenir parelles ($s\_t, a\_t$) d'estat-acció de jugadors experts.
    \item \textbf{Supervisió}: Entrenar una xarxa neuronal per predir l'acció experta $a\_t$ donat l'estat vectoritzat $\mathbf{o}\_t$ (usant el \texttt{StateDUMMY\_UNDERSCOREVectorizer} ja implementat i la loss de cross-entropy).
    \item \textbf{Fine-tuning amb RL}: Utilitzar el model pre-entrenat com a punt de partida per a les fases de RL, combinant el coneixement humà amb l'aprenentatge autònom.
\end{enumerate}

Aquesta aproximació podria accelerar significativament la convergència del RL: en lloc de descobrir des de zero que ``un atac de tipus Aigua és efectiu contra un Pokémon de Foc'', l'agent partiria d'un \textit{prior} expert, i el RL només hauria de refinar les decisions en situacions ambigües o no representades al dataset. 

\subsection{Full de Ruta Immediat: Behavioral Cloning}
\label{subsec:roadmap_bc}

Com a pas següent al RL pur, s'ha començat a preparar l'entorn per a l'\textbf{Imitation Learning}. L'objectiu és utilitzar els replays de jugadors humans d'alt nivell (disponibles a les bases de dades de Showdown) per pre-entrenar la política de l'agent. 

La implementació requerirà:
\begin{enumerate}
    \item Un script de \textit{scraping} per descarregar milers de replays en format JSON.
    \item L'adaptació del \texttt{StateDUMMY\_UNDERSCOREVectorizer} per processar estats històrics a partir dels logs de text.
    \item L'entrenament d'una xarxa de política usant una pèrdua de \textit{Cross-Entropy} per minimitzar la diferència entre l'acció de l'humà i l'agent.
\end{enumerate}
